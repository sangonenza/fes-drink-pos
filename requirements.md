# ロックフェス ドリンク会計・在庫管理ソフト 要件定義 v1.6

作成日: 2026-07-16(v1.6: 2販売所(坂下/坂上)対応 / v1.5: 生ビールを使い捨てカップ実数基準に変更・レジの残数表示を廃止 / v1.4: ソフトドリンクとノンアルを別カテゴリに分離し4分類化 / v1.3: 2日間開催に対応=日別集計・レジ点検を当日基準に / v1.2: 在庫照合ミス防止の仕組みを追加 / v1.1: カテゴリ区分を変更、生ビールの在庫単位を定義)

## 決定事項

| 項目 | 決定 |
|---|---|
| 端末構成 | 複数台(スマホ/タブレット)を**クラウド方式で同期** |
| バックエンド | Supabase 無料枠(Postgres + Realtime) |
| フロントエンド | 静的Webアプリ。Vercel / GitHub Pages 等でホストし、各端末はURLを開くだけ |
| お釣り計算 | **あり**(現金会計時に預かり金額を入力→お釣りを大きく表示) |
| 会計取消 | **あり**(履歴から取消。在庫も自動で戻る) |
| PayPay | 店頭QRを客が読む運用。アプリは支払方法の**記録のみ**(API連携なし) |

## 前提・運用環境

- **2日間開催のイベント**。在庫は1日目から2日目へ持ち越し、売上は日別に集計する
- **販売所は2か所(坂下ドリンク / 坂上ドリンク)**。商品・価格・在庫は販売所ごとに独立(共通の飲み物も販売所ごとに別登録)。各端末は設定で自分の販売所を選び、レジ・在庫・棚卸し・レジ点検は自販売所のみを扱う。集計は「全体/坂下/坂上」で切り替え(既定は自販売所)
- フェス会場のドリンク売り場。タッチ操作前提の大きいボタンUI
- 各端末は自身のモバイル回線でクラウドに接続する
- **回線は不安定である前提**で設計する(下記「オフライン耐性」)

## 画面構成(4画面+2サブ画面)

| 画面 | 役割 |
|---|---|
| ① 会計(レジ) | メイン画面。起動時はここ |
| ② 商品・価格設定 | 商品名・カテゴリ・価格の登録/編集 |
| ③ 在庫管理 | 初期在庫の入力、補充、ロス登録、現在庫の確認 |
| ④ 売上集計 | 履歴・集計・取消・CSVエクスポート |
| ⑤ 棚卸し | ③から遷移。実在庫カウントと差異調整 |
| ⑥ レジ点検 | ④から遷移。現金実査の照合・記録 |

画面上部に常時表示: 同期状態インジケータ(オンライン / オフライン・未送信n件)

## 機能要件

### ② 商品・価格設定
- 商品の登録: **商品名 / カテゴリ / 価格(円)**
- カテゴリは4分類:
  - **ソフトドリンク**(複数種): アクエリ、コーラ等
  - **ノンアルコール**(複数種): ノンアルビール等
  - **缶のお酒**(複数種): 氷結、トリスハイボール等
  - **生ビール**(1種): ビールサーバーで注いで提供
- 商品の編集・削除(削除は非表示化 = 論理削除。過去の売上履歴を壊さない)
- 表示順の並べ替え(よく売れるものを上に)

### ③ 在庫管理
- 商品ごとに初期在庫数を入力
- 在庫の単位はカテゴリで異なる:
  - ソフトドリンク / ノンアルコール / 缶のお酒: **本数**(実数を正確に管理できる)
  - 生ビール: **使い捨てカップの実数**。開始時にカップの数を数えて登録し、締めの棚卸しで残カップ数を数えて差し引く(使用数 = 記録上の販売杯数、が成立するはず。ズレ=打ち忘れ・こぼし未記録)。樽の杯数換算は行わない
- 営業中の**補充(在庫追加)**に対応(生ビールはカップの追加分を入力)
- 補充はケース単位の入力補助(「入数 × ケース数」形式)で暗算ミスを防ぐ
- **ロス登録**: 廃棄・こぼし・スタッフ利用を理由付きでワンタップ記録(未記録のロスは全て「原因不明の差異」になるため必須)
- 現在庫の一覧表示。会計完了で自動減算(全端末に反映)
- 在庫僅少の警告表示(閾値: 残10個以下を目安、設定可)
- **在庫0の商品はレジ画面でグレーアウト**(誤販売防止)。ただし**生ビールはレジに残数を表示せず、グレーアウトもしない**(カップ在庫はビールの残量と一致しないため)

### ① 会計(レジ)
1. 商品ボタンをタップ(**複数選択可**、カテゴリごとに色分け)。タップで数量+1、**長押しでその商品の選択を解除**(誤タップ対策)
2. 選択した商品ごとに数量を +/− で設定
3. 合計金額を常時表示
4. 支払方法を選択: **現金 / PayPay**
5. 現金の場合: 預かり金額を入力(1000円・5000円・10000円のクイックボタン+テンキー)→ **お釣りを大きく表示**
6. 確認画面(商品・数量・合計・支払方法・お釣り)
7. 「決済完了」タップ → 売上記録・在庫減算 → レジ画面に戻る
- どの段階でもキャンセルして最初に戻れる

### ④ 売上集計
- 会計履歴(時刻・端末・商品・数量・支払方法・金額)
- **取消機能**: 履歴の会計を取消(status を cancelled に変更)→ 在庫が自動で戻る。取消済みは打ち消し線で残す(監査性)
- 集計: 総売上 / 現金・PayPay別 / 商品別・カテゴリ別の販売数と金額 / 時間帯別販売数(差異発生時刻の絞り込みに使う)
- **日別フィルタ**(2日間開催対応): 「本日 / 各開催日 / 全期間」を切り替え。既定は本日
- CSVエクスポート: 明細は全期間、集計は選択中の日別フィルタを適用

### ⑤ 棚卸し(在庫照合)モード
- 実際に数えた本数を商品ごとに入力する専用画面
- 入力中は**理論在庫を隠す**(ブラインドカウント)。先に理論値が見えると数え方が雑になるため
- 全商品入力後に「理論在庫 vs 実在庫 vs 差異」を一覧表示
- 差異は理由を付けて**調整イベント**として記録(在庫を直接上書きせず、履歴に残す)
- 棚卸し開始時に未同期の端末があれば警告(「未送信の会計がn件あります」)。全端末同期済みを確認してからカウントさせる
- 想定運用: シフト交代時のミニ棚卸し + 締めの本棚卸し
- 生ビールは**残りの使い捨てカップ数**を数えて入力(缶と同じ差異ロジックで「カップ使用数 vs 記録上の販売杯数」の照合になる)

### ⑥ レジ点検(現金実査)
- 「理論現金残(**本日の**現金売上合計。釣り銭準備金があれば点検時に任意で加算入力)vs 実際のレジ金」を照合・記録
- 当日基準にする理由: 2日間開催で1日目終業時に売上金をレジから回収する運用のため、累計基準だと2日目の点検が合わない
- 在庫差異との切り分けに使う: 在庫だけ合わない→打ち忘れ・ロス / 現金だけ合わない→釣り銭ミス / 両方合わない→会計の記録漏れ

## 同期設計(競合しない設計)

- 売上・在庫変動はすべて**追記専用イベント**として記録し、上書き更新をしない
- 現在庫は保存値ではなく**導出値**: `在庫 = Σ在庫イベント(初期+補充−ロス±調整) − Σ販売数(completedのみ)`
- これにより複数端末が同時に会計しても競合が発生しない(取消も status 変更のみで在庫が自動復元される)
- Supabase Realtime で他端末の会計を数秒以内に反映
- **二重登録防止**: 会計IDは端末側で生成(UUID)し、DB側で一意制約。オフラインキューの再送信が重複しても同じ会計は1件しか登録されない(冪等化)。決済完了ボタンはタップ直後に無効化

## オフライン耐性

- 回線断のとき: 会計操作は継続可能。売上は localStorage のキューに退避し、復帰後に自動送信
- オフライン中の在庫表示は「最終同期時点の値(未反映あり)」と明示
- 商品マスタ・価格は端末にキャッシュし、オフラインでも会計フローが完結する

## アクセス制御

- スタッフ共有のパスコード1つで入場(初回のみ入力、端末に記憶)
- 1日イベントの簡易ガードとしてはこれで十分。URL は部外者に共有しない

## データモデル

```
products:      id, name, category('soft'|'nonal'|'can'|'draft'), location(販売所),
               price, sort_order, active(論理削除フラグ)
               ※ soft=ソフトドリンク / nonal=ノンアルコール / can=缶のお酒 / draft=生ビール
               ※ 商品は販売所に属する。在庫は商品単位の導出値なので自動的に販売所単位になる

stock_events:  id, product_id, qty_delta(±), created_at, reason(任意),
               type('initial'|'restock'|'loss'|'adjust')
               ※ loss=廃棄・こぼし・スタッフ利用 / adjust=棚卸しの差異調整

cash_counts:   id, counted_amount(実査額), theoretical_amount(理論残),
               diff, note, created_at(レジ点検の記録)

sales:         id, device_id, location(販売所), items(jsonb: [{product_id, name, qty, unit_price}]),
               payment('cash'|'paypay'), total,
               cash_received, change,
               status('completed'|'cancelled'), created_at, cancelled_at
```

※ items に商品名・単価をスナップショットとして持たせる(後から価格を変えても過去の売上金額が変わらない)

## 非機能要件

- ボタンは大きく、屋外・混雑・片手操作でも押し間違えにくい
- **スマホ(iPhone / Android)のブラウザで完全動作**すること。スマホ縦画面を基準に設計し、タブレット・PCにもレスポンシブ対応
- ホーム画面に追加すればアプリのように全画面で起動できる(PWA相当のメタ設定)
- 設定・在庫画面はレジ画面から分離し、営業中の誤タップを防ぐ
- 依存を最小にし、当日トラブル時に読める規模のコードに保つ

## 2日間の日次運用

- **1日目終業時**: 棚卸し(ミニでも可)→ レジ点検 → 売上金をレジから回収(釣り銭だけ残すか全額回収)
- **2日目開業時**: 釣り銭をセット → レジ点検画面の「釣り銭準備金」欄にその額を入れて点検すれば基準が合う
- 在庫・商品・価格はそのまま持ち越し(操作不要)。売上集計は「本日」表示が既定なので2日目は自動的に2日目分だけが見える
